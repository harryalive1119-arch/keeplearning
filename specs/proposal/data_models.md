# 数据模型与资源定义

## Intent (设计意图)

为悬疑侦探游戏demo定义统一、可扩展的数据模型和资源结构。确保所有游戏系统（移动、对话、证据收集、谜底揭晓）能够通过数据驱动的方式配置和运行，便于内容迭代和系统扩展。

核心目标：
1. **数据驱动设计**：所有游戏内容通过资源文件定义，无需硬编码
2. **类型安全**：使用Godot的强类型资源和导出属性确保数据完整性
3. **模块化结构**：各系统数据模型相对独立但可相互引用
4. **序列化友好**：支持JSON导入/导出，便于内容创作工具链
5. **版本兼容**：数据结构设计考虑未来扩展的向后兼容性

## Mechanics (逻辑机制)

### 资源管理系统

#### 资源加载策略
1. **按需加载**：场景初始化时仅加载核心资源，其他资源运行时加载
2. **缓存机制**：频繁使用的资源（如角色头像、证据图标）内存缓存
3. **异步加载**：大型资源（如3D模型、音频文件）异步加载避免卡顿
4. **资源池**：可重复使用的资源（如对话气泡、特效）使用对象池

#### 资源标识与引用
1. **全局唯一ID**：所有游戏实体使用字符串ID（如"evidence_knife_001"）
2. **资源路径映射**：通过ID到资源路径的映射表快速定位资源
3. **软引用系统**：资源间引用使用ID而非直接指针，支持热重载
4. **依赖追踪**：自动追踪资源间的依赖关系，确保加载顺序

#### 数据验证与完整性
1. **启动时验证**：游戏启动时检查所有配置数据的完整性和一致性
2. **运行时校验**：关键操作前验证数据有效性，提供友好错误信息
3. **数据迁移**：版本升级时自动迁移旧格式数据到新格式
4. **备份机制**：玩家进度数据自动备份，防止损坏

### 配置数据流程

#### 内容创作流程
```
内容设计师 → 编写JSON/YAML配置文件 → 资源导入工具 → Godot Resource文件 → 游戏运行时加载
```

#### 热重载支持
1. **开发模式**：修改资源文件后自动重新加载，无需重启游戏
2. **实时预览**：内容设计师可在编辑器中实时查看效果
3. **版本控制友好**：纯文本配置文件便于Git等版本控制
